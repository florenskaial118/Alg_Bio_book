11 Основы Linux. Команды.

11.1 Введение: Что такое Linux?

Когда говорят «Линукс», чаще всего имеют в виду не одну программу, а целую экосистему.

  1. Ядро (Kernel): Это самая низкоуровневая программа. Она управляет памятью, устройствами компьютера и распределяет ресурсы между запущенными программами.
  2. Дистрибутив (Distribution): Это ядро + набор системных утилит и интерфейсов. Вместе они образуют готовую операционную систему.

Типы дистрибутивов:

  • Для общего пользования: Есть графический интерфейс. Примеры: Linux Mint, Debian, Ubuntu, российские Астра Linux и Альт Linux.
  • Серверные версии: Графической оболочки нет, только консоль. Это повышает эффективность (серверу не нужно тратить ресурсы на «красивую картинку») и безопасность. Примеры: CentOS, Red Hat (коммерческий), Ubuntu Server, Debian Server.
  • Специализированные: Для узких задач. Например, Kali Linux для тестирования безопасности.

Как мы с ним работаем:

Мы запускаем программу «Терминал» (это окошко на экране). Внутри терминала запускается программа «Шелл» (чаще всего Bash), которая показывает приглашение командной строки (значок $) и ждет наших указаний.

11.2 Философия Linux

11.2.1 Всё — файл

Это ключевая идеология. Файлы, папки, устройства, информация о процессах и настройки драйверов — всё представлено в виде файлов. Это дает универсальный способ доступа ко всей системе через операции чтения и записи.

Посмотрим на устройства, которые представлены как файлы в папке /dev:

ls /dev | head -n 5
afsc_type5
auditpipe
auditsessions
autofs
autofs_control

11.2.2 Конвейерная обработка (Pipelines)

Вертикальный слеш (|) позволяет передавать результат работы одной команды на вход другой. Это позволяет собирать сложные запросы из простых кирпичиков.

Например, найдем все устройства, в названии которых есть pty, а потом посчитаем их количество.

# Ищем совпадения с "pty" в выводе команды ls /dev
ls /dev | grep pty | head -n 5
ptyp0
ptyp1
ptyp2
ptyp3
ptyp4
# Считаем количество таких устройств
ls /dev | grep pty | wc -l
     128

11.2.3 Перенаправление ввода/вывода

Мы можем направлять вывод команд не только в консоль, но и в файлы или переменные.

Запись в файл:

# Создаем папку для примеров (если её нет)
mkdir -p DevOps

# Вывод команды ls сохраняем в файл
ls /dev > DevOps/dev.txt
head -n 3 DevOps/dev.txt
afsc_type5
auditpipe
auditsessions

Запись в переменную:

data=$(cat DevOps/dev.txt)
echo "$data" | head -n 3
afsc_type5
auditpipe
auditsessions

Важно про символы > и >>:

  • > — создает файл и записывает в него. Если файл уже был, он будет перезаписан.
  • >> — если файла нет, создает его. Если файл есть, дозаписывает в конец. Удобно для сбора логов.

# Создадим пустой файл
touch DevOps/test1/empty.txt

11.4 Потоки ввода, вывода и ошибок

В Linux у каждой программы есть три стандартных потока:

  • stdin (0) — входные данные (обычно с клавиатуры).
  • stdout (1) — вывод результата (обычно на экран).
  • stderr (2) — вывод ошибок (обычно на экран).

Их можно перенаправлять.

  1. Перенаправляем только успешный результат (stdout) в файл:
# Успешная команда: вывод уйдет в файл
ls -la DevOps/test1/hello.txt > DevOps/test1/out.txt
cat DevOps/test1/out.txt
-rw-r--r--@ 1 lidaflorenskaya  staff  6 May 14 18:06 DevOps/test1/hello.txt
  1. Ошибка (stderr) не перенаправляется:
# Ошибочная команда: файл out_error.txt будет пустым, а ошибка уйдет в консоль
ls -la DevOps/test2 > DevOps/test1/out_error.txt

11.5 Логические операторы (&& и ||)

Можно строить цепочки команд в зависимости от того, успешно ли выполнилась предыдущая команда (код возврата 0 - успех, или не 0 - ошибка).

  • && (И): Выполнить команду справа, только если команда слева завершилась успешно.
  • || (ИЛИ): Выполнить команду справа, только если команда слева завершилась с ошибкой.
# Попробуем создать папку. Она уже есть, поэтому создание не удастся, и мы НЕ перейдем в нее.
mkdir DevOps/test1 && cd DevOps/test1
pwd  # Мы остались в той же папке
mkdir: DevOps/test1: File exists
/Users/lidaflorenskaya/Documents/АлгБио/1_sem/1_sem_algbio_book
# Команда mkdir завершится ошибкой (папка есть), поэтому сработает команда после ||
mkdir DevOps/test1 || cd DevOps/test1
pwd  # Мы перешли!
mkdir: DevOps/test1: File exists

Это удобно для простейшей автоматизации: например, скачать файл из первого источника, а если он недоступен — из второго.

Изменение прав (chmod):

# Создадим файл для примера

-rwxr-xr-x@ 1 lidaflorenskaya  staff  5 May 14 18:06 DevOps/test1/example.txt

Цифровой способ (chmod 777):

r=4, w=2, x=1. Сумма дает цифру для каждой категории:

  • chmod 755 = rwxr-xr-x (часто для папок)
  • chmod 664 = rw-rw-r-- (часто для файлов)
  • 777 = rwxrwxrwx — не рекомендуется. Это дает полный доступ всем.

Смена владельца (chown): Права может менять только root или владелец файла. Сменить владельца можно командой chown.

Важно: Если на папке нет права x, вы не сможете в неё зайти, даже если есть право r (читать список файлов).

Umask: Это маска, которая определяет, какие права будут отняты у новых файлов и папок по умолчанию.

  • Для файлов базовое значение — 666 (rw-rw-rw-)
  • Для папок — 777 (rwxrwxrwx) umask вычитается из этих значений. Например, umask 002 приведет к тому, что новый файл получит права 666-002 = 664 (rw-rw-r--).

11.7 Итоги и рекомендации

  1. Всегда проверяйте местоположение (pwd) перед массовыми операциями (удаление, перемещение), чтобы не сделать случацно серьезную ошибку
  2. Используйте Tab для автодополнения имен файлов и команд. Это экономит время и предотвращает опечатки.
  3. Принцип минимальных привилегий: Давайте файлам только те права, которые им нужны. Не используйте chmod 777. Для совместной работы используйте группы.
    • Для конфигов с паролями: 600 или 640.
    • Для папок: 755.
  4. Понимание потоков и прав доступа — база для работы с Docker, где эти механизмы используются для изоляции приложений.

12 Bash скрипты

12.1 Введение в Bash скриптинг

В прошлый мы изучили отдельные команды Linux. Но для автоматизации рутинных задач (ежедневное копирование, очистка временных файлов, мониторинг, развертывание приложений) нужно уметь объединять команды и создавать логические конструкции. Для этого существуют bash-скрипты.

Скрипт — это обычный текстовый файл, содержащий последовательность команд и управляющие конструкции (условия, циклы, переменные). Он позволяет:

  • Добиться повторяемости — всегда одинаковый результат.
  • Документировать шаги — скрипт сам описывает последовательность действий.
  • Повысить надежность — уменьшается риск человеческой ошибки.

12.1.1 Структура скрипта

В первой строке скрипта указывается шебанг (shebang) — путь к интерпретатору (#!/bin/bash). Это не всегда обязательно (если скрипт выполняется в текущем shell), но является хорошей практикой.

Создадим наш первый скрипт:

# Создаем директорию для скриптов, если её нет

# Записываем содержимое в файл через cat (альтернатива редактору)
cat > DevOps/scripts/cs1.sh << 'EOF'
#!/bin/bash
echo "Hello world!"
date
pwd
ls -la
EOF

13 Git — система контроля версий

13.1 Зачем нужен Git

Пусть у нас есть курсовая работа. Со временем появляются версии: kursovaya_1, kursovaya_2, kursovaya_final, kursovaya_final_corrected, kursovaya_uzhe-tochno. Знакомо?

Это работает, пока жесткий диск не сломался. А еще попробуйте разобраться в такой папке через месяц или работать над одним файлом втроем.

Git решает эти проблемы. Это система контроля версий — как машина времени для кода и документов. Она позволяет:

  • Отслеживать, кто, когда и зачем менял файлы
  • Откатываться к любой версии
  • Работать над одним проектом параллельно, не затирая изменения друг друга
  • Безопасно экспериментировать, не ломая основную версию

Git создал Линус Торвальдс для разработки ядра Linux. Системы контроля версий бывают двух типов:

  • Централизованные — есть главный сервер, и если он упал, история потеряна
  • Распределенные (Git) — у каждого разработчика полная копия репозитория

Ключевые особенности Git:

  1. Распределенность — каждый хранит репозиторий локально
  2. Скорость — почти все операции выполняются локально
  3. Целостность — все данные хешируются (SHA-1), нельзя незаметно подменить

13.2 Установка и настройка

В большинстве дистрибутивов Git ставится стандартным менеджером пакетов:

sudo apt-get install git

Перед началом работы нужно представиться системе — Git подписывает каждое изменение именем и почтой.

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Настройки можно хранить глобально (для всего компьютера) или локально (для конкретного проекта). Почту и имя никто не проверяет — это формальность, но полезная.

Посмотрим глобальные настройки:

user.name=Lida F
user.email=lida.f.2012@gmail.com
init.defaultbranch=main

13.5 .gitignore — что не хранить в репозитории

В проекте много служебных файлов: временные файлы редактора, скачанные библиотеки, логи, локальные настройки, результаты компиляции. В репозитории им не место — они засоряют историю, увеличивают вес и могут случайно утечь с паролями.

Чтобы Git игнорировал такие файлы, создают .gitignore:

cd DevOps/git_sem
cat > .gitignore << 'EOF'
# Temporary files
*.log
*.tmp
*.text

# Dependencies
node_modules/
vendor/

# Local configs and secrets
.env
.secrets

# System files
.DS_Store
Thumbs.db
EOF

git add .gitignore
git commit -m "add gitignore for temp files and dependencies"

Правила можно задавать через шаблоны (звездочки, слеши). Git перестанет замечать все, что под них попадает.

Правильно настроенный .gitignore:

  • Защищает от утечки паролей и ключей
  • Не дает закоммитить тонны мусора
  • Упрощает перенос проекта — на новом месте все нужное подтянется заново
  • Делает репозиторий легким и универсальным

13.6 Ветвление (branching)

Ветки — одна из самых сильных сторон Git. Ветка — это просто указатель на определенный коммит. Они позволяют разрабатывать функциональность изолированно, не мешая основной линии.

Основная ветка обычно называется master или main. Для разработки новых фич создают отдельные ветки.

cd DevOps/git_sem

echo "=== История коммитов ==="

git branch feature/add-greeting
git checkout feature/add-greeting
git branch

echo -e "\n=== Вносим изменения в ветке ==="
echo "Hello from feature branch" >> readme.txt
git add readme.txt
git commit -m "feat: add greeting line from feature branch"
cat readme.txt

echo -e "\n=== Переключаемся обратно на main и проверяем, что изменений нет ==="
git checkout main

Типичная структура веток в проекте:

  • main (или master) — стабильная версия
  • develop — для разработки
  • release/v1.0.0 — подготовка релиза
  • feature/login-page — конкретные фичи
  • hotfix/critical-bug — срочные исправления

Плюсы ветвления — полный порядок, параллельная работа, изоляция экспериментов. Минусы — требует дисциплины, для маленьких команд может быть избыточно.

13.6.1 Конфликты при слиянии

Git обычно сам находит места изменений и корректно вливает ветки. Но если два человека изменили одну и ту же строку в файле, возникает конфликт. Git не знает, чей вариант правильный, и просит разобраться.

В таких случаях Git подсвечивает конфликтные места в файле, разработчик вручную выбирает нужный вариант и создает коммит слияния.

13.7 Правила именования коммитов и веток

  • Используем латиницу и дефисы (в ветках без пробелов)
  • Без капслока
  • Называем конкретно, не fix, а fix-login-button
  • Если есть трекер задач — добавляем номер задачи, например feature/PROJ-123-add-auth

Типичные префиксы для коммитов:

  • init: — начальная инициализация
  • feat: — новая функциональность
  • fix: — исправление бага
  • docs: — изменения в документации
  • style: — форматирование, отступы, точки с запятой
  • refactor: — рефакторинг без изменения функциональности
  • test: — добавление тестов
  • chore: — обновление вспомогательных инструментов

13.9 Отмена изменений

Иногда после коммита понимаешь, что он лишний. Есть несколько способов отмены.

Безопасный способ — git revert. Он создает новый коммит, который отменяет изменения указанного коммита. История сохраняется полностью.

git revert abc123def

Опасный способ — git reset. Он удаляет коммиты из истории. Полезно, если коммит еще не отправили на сервер, но осторожно — можно потерять изменения.

git reset --soft HEAD~1  # отменить последний коммит, изменения оставить
git reset --hard HEAD~1   # отменить последний коммит и изменения

Еще полезная команда — git cherry-pick. Позволяет скопировать отдельный коммит из одной ветки в другую, не сливая ветки целиком.

git cherry-pick abc123def

rm -rf DevOps/git_sem

После этого:

  • Папка перестает быть репозиторием
  • Вся история коммитов, ветки и настройки теряются безвозвратно
  • Все файлы проекта остаются на месте, но Git их больше не отслеживает

13.12 Советы

  1. Всегда проверяйте git status перед коммитом, чтобы случайно не добавить лишнее
  2. Пишите осмысленные сообщения коммитов — они должны отвечать на вопрос что и почему было сделано
  3. Настраивайте .gitignore в самом начале проекта
  4. Не коммитьте секреты и ключи — они навсегда останутся в истории
  5. Если ветка не доделана, не обязательно ее сливать — stash или просто оставить локально
  6. Для больших команд важно договариваться о правилах именования веток и стиле коммитов
  7. Делайте ревью перед слиянием в основную ветку

14 Docker — контейнеризация

14.1 Зачем нужны контейнеры

Проблема, знакомая каждому разработчику: у тебя всё работает, а у тестировщика или заказчика — нет. Причина почти всегда в окружении:

  • Разные версии языков программирования (например, Python)
  • Устаревшие или отсутствующие библиотеки
  • Конфликты зависимостей между проектами на одной машине
  • Разные пути в конфигурационных файлах
  • Сложность подключения нового человека к проекту

Раньше эту проблему решали виртуальными машинами.

14.1.1 Виртуальные машины vs контейнеры

Виртуальная машина (ВМ) — внутри хостовой ОС запускается гостевая ОС со своим ядром, окружением и библиотеками. Плюсы: полная изоляция. Минусы:

  • Занимают много ресурсов (диск, память, процессор)
  • Долгий запуск (минуты)
  • Не любая ОС совместима

Контейнеры — используют ядро хостовой системы, но изолируют процесс. Создается «песочница» для приложения, которая использует общие ресурсы, но процесс не может выйти за ее пределы.

Преимущества контейнеров:

  • Переносимость — собрав один раз, можно запустить где угодно
  • Изоляция — контейнеры не мешают друг другу и хостовой системе
  • Эффективность — запуск за секунды, минимум оперативной памяти
  • Воспроизводимость — структура описана как код (Dockerfile) и воспроизводится на любой машине

14.1.2 Docker

Docker — это программа для разработки, доставки и запуска приложений в контейнерах. Стал стандартом в мире контейнеризации.

Сравнение ВМ и контейнеров Docker:

Характеристика Виртуальная машина Контейнер Docker
Изоляция Полная (свое ядро) На уровне процесса
Время запуска Минуты Секунды
Совместимость Любая ОС Только с ядром хоста
Ресурсы Много Минимум

14.2 Установка Docker

Скачать Docker можно с официального сайта.

После установки запустим приложение проверим версию и что система готова к работе:

14.3 Архитектура Docker

Docker состоит из нескольких компонентов:

  1. Docker Client — интерфейс взаимодействия с пользователем (команды docker ...)
  2. Docker Daemon (демон) — фоновый процесс, который управляет контейнерами
  3. Registry (хранилище образов) — публичное (Docker Hub) или частное хранилище, откуда можно скачивать и куда загружать образы

14.4 Основные понятия

  • Образ (image) — шаблон контейнера, содержащий приложение и все зависимости
  • Контейнер (container) — запущенный экземпляр образа, имеющий состояние
  • Слой (layer) — каждый шаг в Dockerfile создает слой; слои переиспользуются при сборке
  • Том (volume) — механизм постоянного хранения данных контейнера (чтобы данные не терялись при удалении контейнера)

14.5 Жизненные циклы контейнера

Есть два основных сценария:

  1. Использовать готовый образ из репозитория — docker pull → docker run
  2. Собрать свой образ — написать Dockerfile → docker build

14.6.3 Проброс портов

Порт — это идентификатор сетевого соединения в диапазоне от 1 до 65535. Каждое приложение, которое принимает входящие соединения, «слушает» определенный порт. Например, веб-сервер обычно слушает порт 80.

Конфликт портов возникает, когда два процесса пытаются слушать один и тот же порт одновременно. Операционная система не позволяет этого сделать.

14.7.1 Копирование файлов (docker cp)

docker run -v /путь/на/хосте:/путь/в/контейнере ...

Данные сохраняются даже после удаления контейнера.

14.7.3 Монтирование в Dockerfile (bind mounts)

Bind mount — это способ сделать произвольную папку на хостовой системе доступной внутри контейнера. В отличие от томов, управляемых Docker, bind mount ссылается на абсолютный путь в файловой системе хоста.

# Синтаксис: -v /абсолютный/путь/на/хосте:/путь/в/контейнере
docker run -d \
  -p 8080:80 \
  -v /home/user/project:/usr/share/nginx/html \
  nginx:latest
  • Папка /home/user/project на хосте становится доступна по пути /usr/share/nginx/html внутри контейнера.
  • Изменения, сделанные с любой стороны, мгновенно видны с другой
  • При удалении контейнера данные на хосте остаются нетронутыми

14.9 Многоступенчатая сборка (multi-stage build)

Многоступенчатая сборка позволяет создавать минимальные образы, разделяя этап компиляции (где нужны большие инструменты) и этап выполнения (где нужен только готовый результат).

Например

# Первый этап: сборка
FROM golang:1.20 AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp

# Второй этап: запуск
FROM python:3.10-slim
COPY --from=builder /app/myapp /myapp
LABEL maintainer="your-email@example.com"

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .

EXPOSE 5000
ENV FLASK_APP=app.py
CMD ["python", "app.py"]

14.10 Оптимизация Dockerfile

Для уменьшения количества слоев и размера образа полезно объединять команды, напрмиер в место двух слоев RUN apt-get update RUN apt-get install flask - RUN apt-get update && apt-get install -y flask

14.11 Итоги и рекомендации

Docker решает главную проблему — воспроизводимость окружения на разных машинах. Он легче виртуальных машин, образы собираются из Dockerfile (инфраструктура как код), а реестры обеспечивают распространение образов.

Правила работы с Docker:

  1. Один контейнер — один процесс (не запускать внутри ssh и другие сервисы)
  2. Не хранить секреты в образах (логины, пароли, ключи)
  3. Использовать несколько связанных контейнеров через сети, а не ставить всё в один
  4. Указывать версии образов и зависимостей (не использовать latest в production)
  5. Удалять ненужные образы и контейнеры, чтобы не забивать место
  6. Использовать теги для версионирования образов

16 Управление многоконтейнерными приложениями (Docker Compose)

16.1 Как мы дошли до Compose: от монолита к экосистеме

Сначала были монолиты: всё приложение (веб-сервер, БД, логика) работало в одном модуле. Это создавало трудности с обновлением и отладкой — падение одной части могло остановить всё.

Затем монолиты начали дробить на микросервисы — маленькие, специализированные куски кода. Docker решил проблему изоляции, упаковав каждый микросервис в контейнер.

Но приложение — это не один контейнер, а целая экосистема. Например: веб-сервер, база данных, сервис хэширования. Запускать их вручную — значит:

  • писать длинные команды с кучей флагов для каждого контейнера,
  • соблюдать порядок запуска,
  • помнить про сетевые настройки.

Такую конфигурацию сложно передать коллегам, хранить в гите или разворачивать на другом сервере.

Здесь и появляется Docker Compose — инструмент для описания и запуска многокомпонентных приложений через один конфигурационный файл и одну команду.

16.2 Иерархия сущностей (от простого к сложному)

  1. Docker-контейнер (управляется через командную строку)
  2. Docker Compose (группа контейнеров, описанных вместе)
  3. Оркестрация (управление группами контейнеров в кластерах — Kubernetes, Swarm)

16.3 Преимущества Docker Compose

  • Воспроизводимость — один файл описывает всё окружение.
  • Изоляция — уязвимость в веб-сервере не скомпрометирует базу данных.
  • Упрощение разработки — всё окружение поднимается одной командой.
  • Портативность — конфиг одинаково работает на Linux, macOS и Windows.
  • Версионность — файл Compose можно хранить в Git.

Пример из инета

cd DevOps/docker
cat > compose.yaml << 'EOF'
# Use "services" to define the different parts of your application
services:
  # Service name: 'web'
  web:
    # Build the image from a Dockerfile in the current directory
    build: .
    # Map host port 8000 to container port 5000
    ports:
      - "8000:5000"
    # Set environment variables for the application
    environment:
      - REDIS_HOST=redis
      - REDIS_PORT=6379
    # Ensure 'redis' starts before 'web'
    depends_on:
      - redis

  # Service name: 'redis'
  redis:
    # Pull a pre-built image from Docker Hub
    image: "redis:alpine"
    # Persist data using a volume
    volumes:
      - redis_data:/data

# Define named volumes for persistent storage
volumes:
  redis_data:
EOF

16.4 Структура конфигурационного файла (compose.yaml)

Формат файла критичен к отступам. Основные разделы:

  • services — список сервисов (контейнеров), которые будут запущены.
    • web / database — имя сервиса.
    • image — готовый образ из репозитория (например, nginx:latest).
    • build — указание на кастомный Dockerfile.
    • ports — проброс портов (хост:контейнер).
    • environment — переменные окружения (логины, пароли, хосты).
    • env_file — ссылка на файл с секретами (должен быть в .gitignore).
    • depends_on — указание на зависимые сервисы (порядок запуска).
    • restart — политика перезапуска.
  • networks — описание сетей для изоляции сервисов.
  • volumes — монтирование дисков (персистентные данные).

16.5 Базовые команды для работы

  • docker compose up -d — запустить приложение в фоне (демон).
  • docker compose down — остановить и удалить контейнеры.
  • docker compose ps — посмотреть статус сервисов.
  • docker compose config — проверить синтаксис и вывести итоговый конфиг.
  • docker compose logs — посмотреть логи.
  • docker compose up --scale [сервис]=[количество] — запустить несколько экземпляров сервиса.

16.6 Ключевые концепты

Сеть Compose автоматически создает внутренний DNS. Контейнеры могут общаться друг с другом по имени сервиса (например, redis), а не по IP. Это удобно, так как IP может меняться.

Практика: создавайте несколько сетей (frontend, backend), чтобы изолировать сервисы. Например, база данных может быть в сети backend без выхода в интернет (internal: true).

Тома (volumes) Позволяют сохранять данные между перезапусками. Можно монтировать том в режиме read-only для файлов, которые не должны меняться внутри контейнера.

Переменные окружения: иерархия приоритетов 1. Переменные, заданные прямо в compose.yaml. 2. Значения из файла .env. 3. Переменные из хост-окружения. 4. Значения по умолчанию внутри Docker-образа.

16.7 Политики перезапуска (restart)

Применяются демоном Docker в нештатных ситуациях.

  • always — перезапуск всегда, даже после docker stop. Используется для веб-сервисов и критических микросервисов, которые должны быть доступны 24/7.
  • no — ручное вмешательство при падении. Подходит для тестов, пакетных задач, миграций БД.
  • on-failure:5 — перезапуск при ненулевом коде ошибки (до 5 раз). Используется для обработчиков очередей.
  • unless-stopped — как always, но не перезапускается после ручной остановки (docker stop). Идеально для продакшн-сервисов, БД, где важна возможность остановить для обновления.

16.8 Лучшие практики

  • Храните секреты в файле .env и добавьте его в .gitignore.
  • Используйте healthcheck для проверки реальной работоспособности контейнера (не только depends_on).
  • Ограничивайте ресурсы (deploy: resources: limits) по памяти и CPU, чтобы один сервис не “уронил” весь сервер.
  • Назначайте политики перезапуска осознанно — иногда лучше, чтобы контейнер упал и сообщил об ошибке, чем бесконечно перезапускался в нерабочем состоянии.

16.9 Типичные сценарии использования

  • Запуск многоконтейнерного приложения для разработки.
  • Изолированное тестирование интеграций.
  • Быстрая демонстрация продукта клиенту или на выставке.
  • Продакшн-среда для небольших проектов (хотя для сложных случаев лучше подходит Kubernetes).

17 Введение в DevOps: SSH, Docker Swarm и Ansible

17.1 SSH — безопасное удаленное подключение

Когда мы работаем с виртуальными машинами (ВМ) и серверами, нам часто нужно уметь подключаться к ним удаленно.

17.1.1 Что такое SSH?

SSH (Secure Shell) — это сетевой протокол для безопасного удаленного управления компьютером.

  • Что нужно для подключения: IP-адрес хоста и логин пользователя.
  • Что позволяет делать: удаленно попасть на хост, безопасно копировать файлы.
  • Почему это безопасно: весь трафик шифруется.

17.1.2 Как работают ключи SSH (простое объяснение)

SSH использует пару ключей: закрытый (private key) и публичный (public key).

  • Закрытый ключ — это ваш секрет. Он остается только у вас и никому не передается.
  • Публичный ключ — вы кладете его на сервер, к которому хотите подключаться.

Как это работает:

  1. Вы генерируете пару ключей командой ssh-keygen. В папке .ssh появляются два файла: id_rsa (закрытый) и id_rsa.pub (публичный).
  2. Вы копируете публичный ключ на удаленный сервер.
  3. Когда вы подключаетесь, сервер проверяет, что у вас есть соответствующий закрытый ключ.
  4. Если ключи подходят, подключение происходит без ввода пароля.

17.1.3 Основные команды SSH

Подключение к серверу:

ssh server@log
# где server — логин, log — адрес сервера (или IP)

17.2 Docker Swarm — оркестрация контейнеров

17.2.1 Что такое оркестрация?

Оркестрация — это объединение нескольких хостов (машин) в единый кластер для управления контейнерами.

Типы инструментов:

  1. Обычный Docker (один хост)
  2. Docker Compose (несколько контейнеров на одном хосте)
  3. Docker Swarm / Kubernetes (кластер из нескольких хостов)

17.2.2 Зачем нужна оркестрация?

Задача Решение
Отказоустойчивость При падении одной ноды (сервера) нагрузка переключается на другую
Балансировка нагрузки Запросы распределяются между несколькими серверами
Эластичность Можно добавлять мощности, когда нужно, и уменьшать, когда не нужно

17.2.3 Архитектура Swarm

  • Менеджеры (managers): управляют кластером, принимают решения.
  • Воркеры (workers): выполняют рутинные задачи (запускают контейнеры).

Правило для отказоустойчивости: нужно нечетное количество менеджеров (3, 5, 7…). Это позволяет кластеру продолжать работу при отказе части менеджеров.

17.2.4 Конфигурация (docker-stack.yml)

Это файл в формате YAML, который описывает архитектуру кластера. Он похож на docker-compose.yml, но для кластера.

Что описывается:

  • Сервисы (services) — что запускаем.
  • Имена и образы (images) контейнеров.
  • Конфигурации, права, порты.
  • Обновление (update): механизм обновления нод (параллельно или последовательно).

17.2.5 Основные команды Docker Swarm

Инициализация Swarm (создание кластера):

docker swarm init --advertise-addr 192.168.0.22

После этой команды текущий узел становится менеджером.

Получение токена для подключения воркера:

docker swarm join-token worker

Токен — это секретный код. Он дает возможность подключаться к кластеру. Храните его в безопасном месте!

Подключение новой ноды (воркера):

docker swarm join --token <токен>

Подключение нового менеджера:

docker swarm join-token manager
# получаем токен для менеджера
docker swarm join --token <токен-менеджера>

Выход из кластера:

docker swarm leave

Повышение воркера до менеджера:

docker node promote <имя-воркера>

Понижение менеджера до воркера:

docker node demote <имя-менеджера>

17.2.6 Управление сервисами

Масштабирование (увеличение числа реплик):

docker service scale <сервис>=5

Это увеличивает количество запущенных копий сервиса до 5 без остановки работы.

Обновление сервиса:

docker service update

Откат (rollback): можно проверить новую версию на одном экземпляре, а потом раскатить на весь кластер. Если что-то пошло не так — откатиться.

Ограничения ресурсов: можно задать лимиты на CPU и память для каждого контейнера (в конфиге или командной строке).

17.2.7 Обязательные поля в конфиге docker-stack.yml

Поле Что означает
version Версия Docker Engine
ports Порты внутри контейнера и на кластере для связи
mode Режим связи (например, host или ingress)
replicas Количество реплик (копий сервиса)
rollback_config Политика отката
secrets Пароли, доступы к базам данных (хранятся в зашифрованном виде)
networks Своя локальная сеть между нодами, защищенная от вмешательства

17.2.8 Управление стеком (stack)

Удаление всего стека:

docker stack rm stack-name

Нельзя удалить отдельные контейнеры — удаляется всё целиком.

Запуск стека:

docker stack deploy --compose-file docker-stack.yml stack-name

Просмотр состояния сервисов:

docker service ls

17.2.9 Типовой пример стека

В примере могут быть: nginx (веб-сервер), health check (проверка здоровья), API, Redis (кэш), Prometheus (мониторинг), Grafana (дашборды).

17.3 Ansible — автоматизация конфигурации

17.3.1 Что такое Ansible?

Ansible — это инструмент для автоматизации настройки серверов. Все подключения идут через SSH.

Ключевые особенности:

  • Идемпотентность: повторный запуск не ломает то, что уже настроено. Если система уже в нужном состоянии — ничего не меняется.
  • Агент не нужен: не требуется устанавливать дополнительное ПО на управляемые серверы (нужен только Python и SSH-сервер).
  • Конфигурация в явном виде: описание хранится в репозитории, легко читается и понимается.

17.3.2 Как работает Ansible

  1. Есть управляющий узел (где установлен Ansible).
  2. Ansible подключается к серверам по SSH.
  3. Копирует на сервер небольшой модуль (временный файл).
  4. Выполняет нужные действия.
  5. Удаляет временные файлы.

Требования к управляемым узлам: только Python и SSH-сервер.

17.3.3 Установка Ansible

pip install ansible
ansible --version

17.3.4 Структура типового Ansible-проекта

project/
├── ansible.cfg           # конфигурация Ansible
├── group_vars/           # переменные для групп хостов
├── host_vars/            # переменные для отдельных хостов
├── inventory/            # список серверов
│   ├── production/       # продакшен-окружение
│   └── staging/          # тестовое окружение
├── playbooks/            # плейбуки (сценарии)
├── roles/                # роли (переиспользуемые блоки)
├── requirements.yml      # зависимости (роли из galaxy)
└── site.yml              # главный плейбук

17.3.5 Настройка ansible.cfg

Что можно настроить:

  • Путь к inventory (списку хостов)
  • Путь к ролям (roles_path)
  • Путь к SSH-ключам (чтобы не вводить пароль)
  • Настройки повышения привилегий (privilege escalation)

17.3.6 Inventory — описание серверов

В inventory мы описываем, какие сервера у нас есть и за что они отвечают.

Формат (может быть YAML или INI):

web-servers:
  hosts:
    web-01:
      ansible_host: 192.168.1.10
    web-02:
      ansible_host: 192.168.1.11
  vars:
    ansible_user: ubuntu

db-servers:
  hosts:
    db-01:
      ansible_host: 192.168.1.20

Что можно указать:

  • Адреса серверов
  • Имена внутри системы
  • Пользователя для подключения
  • Переменные для работы

17.3.7 Проверка доступности хостов

# Проверить все хосты
ansible all -m ping

# Проверить группу webserver
ansible webserver -m ping

# Посмотреть настройки конкретного хоста
ansible-inventory --host web-01

17.3.8 Выполнение команд на удаленных машинах

Модуль command (безопасный, без shell):

ansible all -m command -a "hostname"

Модуль shell (гибкий, но опасный):

ansible all -m shell -a "hostname && uptime"

Можно объединять команды, но есть риск навредить системе.

Копирование файлов:

ansible all -m copy -a "src=/my/local/file dest=/remote/path/"

17.3.9 Playbook — главный инструмент Ansible

Playbook — это конфиг с желаемым состоянием системы. Ansible сам разбирается, как его достичь.

Пример структуры playbook:

- name: Настройка веб-серверов
  hosts: webservers
  become: yes
  roles:
    - nginx
    - certbot

17.3.10 Роли (roles)

Роль — это переиспользуемый блок, который описывает настройку одного компонента (например, nginx, PostgreSQL, Redis).

Структура роли:

roles/nginx/
├── defaults/     # переменные по умолчанию
├── files/        # статические файлы
├── handlers/     # обработчики (например, перезапуск сервиса)
├── meta/         # метаданные роли
├── tasks/        # основные задачи
├── templates/    # шаблоны (.j2)
└── vars/         # переменные роли

Пример tasks/main.yml:

- name: Установка nginx
  apt:
    name: nginx
    state: present
    update_cache: yes

- name: Настройка конфига
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    validate: nginx -t
  notify: reload nginx

17.3.11 Запуск playbook

# Основной запуск
ansible-playbook ./playbooks/webservers.yml

# С подробным выводом (-v, -vv, -vvv)
ansible-playbook playbook.yml -vv

# Запуск только на определенных хостах
ansible-playbook playbook.yml --limit 'webservers:!web-02'

# Запуск с конкретной задачи
ansible-playbook playbook.yml --start-at-task "Включить виртуальные хосты"

# Запуск только задач с определенными тегами
ansible-playbook playbook.yml --tags "install,config"

17.3.13 Переменные в Ansible

Иерархия переменных (от меньшего приоритета к большему):

  1. В роли (defaults)
  2. В inventory (сначала групповые, потом для конкретного хоста)
  3. В playbook
  4. В таске
  5. Из командной строки (экстра-переменные) — наивысший приоритет

Динамические переменные (вычисляются на лету):

- set_fact:
    deploy_path: "/var/www/{{ app_name }}-{{ version }}"

17.3.14 Шаблоны (Jinja2)

Файлы с расширением .j2 позволяют использовать переменные и логику внутри конфигов.

17.3.15 Handlers (обработчики)

Handlers — это задачи, которые выполняются только один раз в конце, даже если их вызвали несколько раз через notify. Они запускаются только если задача с notify действительно что-то изменила.

Пример:

- name: Обновить конфиг nginx
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: reload nginx

handlers:
  - name: reload nginx
    service:
      name: nginx
      state: reloaded

17.3.16 Теги (tags)

Теги позволяют запускать не весь playbook, а только конкретные задачи.

ansible-playbook playbook.yml --tags "config"

17.3.17 Ansible Vault — шифрование секретов

Чтобы хранить чувствительные данные (пароли, ключи) в Git, их можно зашифровать.

# Зашифровать файл
ansible-vault encrypt group_vars/all/secrets.yml

# Расшифровать
ansible-vault decrypt group_vars/all/secrets.yml

# Запустить playbook с запросом пароля
ansible-playbook playbook.yml --ask-vault-pass

17.3.18 Ansible Galaxy — публичный реестр ролей

Можно использовать готовые роли от сообщества.

# Установка роли из Galaxy
ansible-galaxy install geerlingguy.nginx

# Установка всех зависимостей из requirements.yml
ansible-galaxy collection install -r requirements.yml

17.3.19 Важное замечание про shell-команды

При выполнении команд через модуль shell или command Ansible всегда считает, что состояние системы изменилось, потому что не знает, изменилось ли что-то на самом деле.

Лучшая практика: использовать готовые модули (apt, service, copy, template и т.д.), а не raw shell-команды. Модули умеют проверять текущее состояние и меняют только то, что нужно.

17.4 Итоговая архитектура Ansible-проекта

  1. Папка проекта с ansible.cfg.
  2. Папка inventory — список серверов, сгруппированных по признакам.
  3. Папка playbooks — YAML-файлы с описанием того, что делать.
  4. Папка roles — переиспользуемые роли (nginx, postgresql и т.д.).
  5. Папка group_vars / host_vars — переменные для групп и отдельных хостов.

Важно: в Ansible нет встроенного механизма отката (rollback). Единственный способ — вернуться к предыдущей версии конфигурации через Git.

17.4.1 Главный принцип Ansible

Идемпотентность: если запустить playbook на уже настроенном сервере, ничего не должно сломаться. Система всегда будет приведена к тому состоянию, которое описано в конфиге. Если сервер уже в нужном состоянии — изменений не будет.

Это позволяет:

  • Делать одинаковые конфигурации на всех серверах.
  • Гарантировать, что все серверы в кластере одинаковы.
  • Быстро находить и исправлять серверы с “плохим” конфигом.

18 Теория по CI/CD

18.1 Зачем нужен CI/CD

Представьте конец рабочего дня, и нужно сделать горячий фикс. Разработчик исправляет одну строчку, локальные тесты зелёные, он вручную деплоит изменение. Но на проде оказывается другая версия Node.js, и всё ломается. Сервис падает, метрики краснеют, нужен откат. Это не ошибка разработчика — это проблема плохого CI/CD.

Рассмотрим диаграмму ручного процесса: - решение задачи и коммит - дни тишины (ожидание проверки коммита) - мёрж в основную ветку - ручная сборка - ручной деплой - падение прода

Каждый шаг — это место, где человек может ошибиться. И даже если он не ошибётся сейчас, это может произойти в следующий раз. Именно здесь помогает CI/CD система.

18.2 Как работает автоматический конвейер

Процесс в CI/CD выглядит так: - коммит - автоматическая сборка проекта - автоматические тесты - стейджинг (staging) - проверка здоровья (health check) - прод

Ошибка будет обнаружена в течение 5–7 минут после коммита, а система продолжит работать без изменений. Маленькие изменения несут маленький риск, команда перестаёт бояться кнопки «релиз» даже в пятницу. Благодаря автоматизации процесс становится надёжнее. Плюс, есть артефакты — предыдущие работающие сборки, и в случае падения можно легко откатиться на прошлую версию.

18.3 Continuous Integration (непрерывная интеграция)

  • запускается на каждый push или pull
  • включает линтер (статический анализатор кода), юнит-тесты, сборку артефакта
  • должен укладываться в 5–10 минут
  • правила:
    • запрещён мёрж в main с красным CI
    • ветка живёт меньше одного дня
    • каждый коммит проходит сборку — даже «фикс опечатки»

18.4 Continuous Delivery vs Continuous Deployment

Параметр Delivery Deployment
Деплой на стейджинг автоматический автоматический
Деплой на прод по кнопке Approve автоматически, без кнопки
Роль человека постоянно проверяет результаты тестов не вмешивается, если тесты зелёные
Требования — надёжные тесты и авто-откат по метрикам
Примеры банки, медицина, госсектор (высокая цена ошибки) Netflix, Vercel, внутренние инструменты, стартапы (важна скорость доставки)

Аналогия: delivery — посылка упакована и лежит на почте, отправляется по звонку начальника. deployment — посылка сама запрыгивает на почту и уезжает.

18.5 Что и когда выбирать

Delivery выбирают, если: - есть законодательные ограничения - клиент требует подпись перед релизом - команда не готова к автоматике - релизы редкие (например, проект в конце жизненного цикла)

Deployment выбирают, если: - есть надёжные тесты и авто-откат по метрикам - релизы частые - небольшая команда доверяет автоматике - используются feature flags

Для реализации Delivery в инструментах: - GitLab CI: when: manual - GitHub Actions: environment: production

Без этих параметров получится Deployment.

18.6 Ключевые практики (7 классических)

  1. Одна кодовая база на сервис
  2. Автоматическая сборка — без ручных установок и очисток, среда едина для всех
  3. Авто-тесты: юнит, интеграционные, линтеры. Конспект отмечает, что многими пренебрегают, но сейчас меняется законодательство — появляются требования по статическому и динамическому анализу кода, просмотру уязвимостей в библиотеках
  4. Короткие ветки — меньше одного дня (меняется небольшой кусок кода, проще тестировать мелкие коммиты)
  5. Окружение, похожее на прод — те же версии баз данных, переменные окружения
  6. Build once — deploy everywhere (один артефакт разворачивается везде: тестовая, продуктовая среды)
  7. Мониторинг после деплоя — хотя бы примитивная проверка жизнеспособности поднятых сервисов

18.7 Эволюция CI/CD инструментов

Поколение 1 (2000-е): Jenkins, TeamCity — bash-скрипты

Поколение 2 (2015–2020): GitLab CI, GitHub Actions — YAML-конфиги, встроенные реестры (registry), но много копипасты

Поколение 3 (2022–2025): Earthly, Dagger, Nix, Renovate — декларативные, работают локально, поддерживают reuse

18.7.0.1 Современные практики

  • Пайплайны as code с возможностью переиспользования (reuse)
  • Декларативная сборка — Earthly, Dagger, Nix вместо bash-скриптов (работает одинаково везде)
  • Авто-обновление зависимостей — Dependabot, Renovate (можно ничего не знать об их работе, они сами обновляют библиотеки)
  • Безопасные цепочки поставок — SBOM (список библиотек с версиями), SLSA (уровни подтверждения, что артефакт собран именно так, как мы думаем)
  • AI в CI/CD — Copilot сам пишет пайплайны и отлаживает тесты
  • OIDC вместо статических секретов — временные токены (например, на 1 час)

18.8 OIDC: почему это важно

Раньше использовали статические ключи. AWS-ключ хранится в секретах CI годами. Если украли — используют неделями. Ротация ключей — ручная боль. Нет привязки к конкретному пайплайну.

Современная схема пайплайна: - коммит - юнит-тесты и проверки (checks) - тесты + сборка - push в registry - deploy на стейджинг (доставка по SSH и разворачивание) - smoke-тесты - approve и прод

18.9 Reusable workflows — DRY для пайплайнов

Проблема: 50 репозиториев, 50 копий одного пайплайна.

Решение: reusable workflow в GitHub Actions.

Пример:

name: Reusable Build
on:
  workflow_call:
    inputs:
      ...
jobs:
  build_and_push:
    runs-on: ...
    - uses: actions/attest-sbom@v1

Вызов в других репозиториях

name: CI
on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  # вызов переиспользуемого workflow
  call-build:
    uses: your-org/shared-workflows/.github/workflows/reusable-build.yml@main
    with:
      environment: staging
      node_version: '20'
    secrets:
      DOCKER_USERNAME: ${{ secrets.DOCKER_USERNAME }}

18.10 Earthly — makefile для контейнеров

  • синтаксис как у Dockerfile — низкий порог входа
  • один файл работает одинаково локально и в CI
  • цели: test, build, docker
  • умное кеширование на уровне целей
  • параллельное выполнение независимых целей
  • хорошая воспроизводимость — фиксирует версии инструментов
  • идеален для монорепозиториев

Минус: требует Docker и в очень сложных сценариях может быть менее гибким, чем полноценный язык программирования.

18.11 Dagger

  • пайплайн на Go, Python, TypeScript — никакого YAML
  • можно запускать локально и на любом CI/CD
  • можно писать юнит-тесты на сам пайплайн
  • можно ставить брейкпоинты, запускать дебаггер

Выбирать, если: - сложная логика ветвлений - команда знает Go/Python/TS

Минус: мало готовых решений — придётся писать самому.

18.12 Nix

  • функциональный менеджер пакетов
  • то, что собралось сегодня, соберётся так же через год
  • ключевая идея: каждая сборка описывается чистой функцией, которая получает зависимости и выдаёт артефакт — никаких сюрпризов от необновлённых библиотек

Минус: высокий порог входа.

Все три инструмента можно комбинировать между собой.

18.13 Безопасность цепочки поставок: SBOM, SLSA

SBOM (Software Bill of Materials) — список компонентов приложения: - библиотеки, их версии - список транзитивных зависимостей - инструменты сборки - слои

Зачем нужен SBOM: - вышла новая уязвимость — за 5 минут находишь все сервисы, где используется зависимость - юридические требования (например, GPL-зависимость требует аудита лицензий) - крупные заказчики требуют SBOM на тендерах - у Минцифры есть списки рекомендованного и нерекомендованного ПО, а также импортозамещённого

Инструмент: actions/attest-sbom@v1

SLSA отвечает на вопросы: - как и откуда собран артефакт? - из какого коммита и репозитория? - какой пайплайн и окружение использовались? - не был ли артефакт подменён после сборки?

Инструменты: cosign, slsa-github-generator

18.14 Авто-обновление зависимостей

Два основных инструмента: Renovate и Dependabot.

Dependabot: - встроен в GitHub, настраивается за 5 минут - бесплатно для public-репозиториев - проверяет зависимости, проводит тесты, если прошли — авто-мёрж, иначе создаёт задачу

Минусы: только для GitHub, один PR на одно обновление, меньше гибкости.

Renovate: - работает на всех платформах: GitHub, GitLab, Bitbucket, Azure DevOps, self-hosted - создаёт issue-панель со всеми обновлениями - сложнее настройка, но гораздо мощнее Dependabot

18.15 Четыре метрики оценки качества пайплайна (DORA)

Для уровня:

  • Lead Time for Changes — менее 1 часа (время от первого коммита до деплоя на прод)
  • Deployment frequency — несколько раз в день (как часто успешно деплоите в продакшн)
  • Mean Time To Recovery ( MTTR - среднее время восстановления после инцидента) — менее 1 часа
  • Change failure rate — 0–5% (процент деплоев, вызвавших инцидент)

18.16 Антипаттерны

  1. Пайплайн на 45 минут — никто не ждёт, CI всегда красный. Решение: вынести долгие тесты в отдельный stage.
  2. Flaky test (нестабильный тест) — тест, который падает без реальной причины.
  3. Нет отката (rollback) — после неудачного деплоя невозможно быстро вернуться к предыдущей версии.
  4. Долгоживущие секреты в CI.

18.17 Четыре столпа современного CI/CD (по итогам занятия)

  1. Reuse (переиспользование)
  2. Декларативность
  3. Безопасность
  4. Автоматизация зависимостей

19 Terraform: инфраструктура как код

19.1 Что такое Infrastructure as Code (IaC)?

IaC — это управление инфраструктурой (серверы, сети, диски, базы данных) через код, а не через ручные действия (консоль или GUI).

Плюсы IaC:

  • Воспроизводимость
  • Контроль версий (Git)
  • Автоматизация тестирования и развертывания
  • Снижение человеческих ошибок

Terraform — это декларативный инструмент для IaC от HashiCorp. Он описывает желаемое состояние инфраструктуры на языке HCL, строит граф зависимостей, сравнивает желаемое состояние с текущим (state) и приводит инфраструктуру к нужному виду.

19.2 Декларативный vs императивный подходы

Декларативный (Terraform) Императивный (ручные скрипты или Ansible плейбуки)
Что должно быть? Как это сделать?
Инструмент сам ищет путь к цели Разработчик управляет порядком команд
Идемпотентность из коробки Требует ручной идемпотентности
Предсказуемый результат Хрупкость при изменении окружения

19.3 Место Terraform среди других инструментов

  • AWS CloudFormation — только AWS, YAML/JSON, нет внешнего управления состоянием (state)
  • Pulumi — использует обычные языки (Python, TypeScript, Go)
  • Ansible — конфигурация ОС и приложений, не provisioning инфраструктуры
  • Terraform — универсальный (AWS, GCP, Azure, Kubernetes, DNS, 100+ провайдеров)

Главное преимущество Terraform: единый workflow для любого облачного провайдера.

19.4 Terraform vs Ansible (не конкуренты, а партнеры)

Terraform Ansible
Декларативный В основном императивный (плейбуки)
Создает и управляет облачными ресурсами Настраивает ОС и приложения
Хранит state (знает прошлое) Не хранит state
Изменяет инфраструктуру на каждом запуске Можно запускать много раз идемпотентно

Лучшая практика: Terraform создает инфраструктуру (ВМ, сети) и передает IP-адреса в Ansible, который уже настраивает ОС и приложения внутри.

19.5 Базовые команды Terraform

terraform init          # загружает провайдеров и модули
terraform plan          # показывает, что изменится (обязательно перед apply)
terraform apply         # применяет изменения (спрашивает подтверждение)
terraform fmt           # форматирует код
terraform validate      # проверяет синтаксис HCL

Пример main.tf (провайдер local):

terraform {
  required_providers {
    local = {
      source = "hashicorp/local"
    }
  }
}

resource "local_file" "example" {
  filename = "hello.txt"
  content  = "Hello from Terraform!"
}

19.6 Язык HCL (HashiCorp Configuration Language)

  • Блоки: resource "type" "name" { ... }
  • Аргументы: key = value
  • Комментарии: # или // (однострочные), /* */ (многострочные)
  • Строки, выражения, функции: format, join, file, length, upper

Locals — промежуточные вычисления внутри конфигурации:

locals {
  environment = "prod"
  full_name   = "${local.environment}-server"
}

Используйте terraform console для отладки выражений.

19.7 Провайдеры и мета-аргументы

  • Провайдер — плагин, который общается с API облака/БД/DNS (например, hashicorp/aws, hashicorp/google, hashicorp/local)
  • count — создаёт N экземпляров ресурса (count = 3)
  • for_each — создаёт экземпляры по ключам map/набора
  • depends_on — явное управление зависимостями (редко нужно, Terraform обычно сам строит граф)

count vs for_each: - count — по индексу (подходит для однотипных ресурсов) - for_each — по ключам map (лучше для изменяемых наборов)

19.8 Входные переменные (input variables)

Позволяют параметризовать конфигурацию для разных окружений.

variable "environment" {
  type    = string
  default = "dev"
  sensitive = true   # не показывать в логах
}

Приоритет (от низшего к высшему):

  1. default
  2. terraform.tfvars (файл с переменными)
  3. переменные окружения TF_VAR_env
  4. флаг -var="env=prod"

Секреты храните в *.tfvars и не коммитьте их в Git (добавьте в .gitignore).

19.9 Выходные значения (outputs)

Используются, чтобы передать важные данные (IP, DNS, ID ресурсов) в другие системы или в Ansible.

output "instance_ip" {
  value = aws_instance.web.public_ip
}

19.10 State (состояние)

terraform.tfstate — JSON-файл, который маппит конфигурацию на реальные объекты в облаке.

Проблемы локального state:

  • Конфликты при командной работе (кто последний сохранил — тот прав)
  • Хранит секреты в открытом виде
  • Можно потерять файл и связь с инфраструктурой

Решение — удаленные бэкенды (backend):

  • S3 (AWS), GCS (Google), Azure Storage, Consul и др.
  • Настраиваются в блоке terraform { backend "s3" { ... } }

Правило: в командном проекте всегда используйте удаленный бэкенд.

19.11 Работа со state (команды)

terraform state list          # список ресурсов в state
terraform state show <ресурс> # подробно о ресурсе
terraform state mv old new    # переименовать ресурс в state
terraform state rm <ресурс>   # удалить из state (не удаляя реальный ресурс)
terraform import <ресурс> <id> # импортировать существующий ресурс в state

19.12 Жизненный цикл и мета-аргумент lifecycle

  • Terraform строит граф зависимостей и создает ресурсы в правильном порядке.
  • При изменении — in-place update (обновление на месте) или destroy + create.
  • create_before_destroy = true — сначала новый, потом старый (минимальный downtime).
  • prevent_destroy = true — защита от случайного удаления.
  • ignore_changes = [tags] — игнорировать изменения определенных полей.
lifecycle {
  create_before_destroy = true
  prevent_destroy       = true
  ignore_changes        = [tags]
}

19.13 Provisioners (используйте с осторожностью!)

Выполняют скрипты на созданном ресурсе (например, remote-exec, local-exec).

Почему их лучше избегать:

  • Нарушают декларативность
  • Не идемпотентны
  • Сложно отлаживать, не видны в terraform plan

Лучше использовать Ansible или cloud-init вместо provisioners.

19.14 Модули

Модуль — контейнер для нескольких ресурсов, используемых вместе. Позволяет переиспользовать и стандартизировать код.

#| eval: false
module "vpc" {
  source = "./modules/vpc"
  env    = var.environment
}
  • Модули версионируют: source = "git::https://...?ref=v1.2.0"
  • Registry модулей: registry.terraform.io

19.15 CI/CD + Terraform

Пример GitLab CI:

stages:
  - validate
  - plan
  - apply
  - destroy

apply:
  stage: apply
  when: manual   # ручное подтверждение

GitHub Actions: запуск plan на pull request, apply на push в main.

Безопасность:

  • Никогда не храните ключи в коде → CI Secrets + OIDC (вместо долгоживущих ключей AWS)
  • Блокировка state (например, DynamoDB для S3 backend) предотвращает параллельные записи
  • Всегда plan перед apply

Best practices:

  • terraform fmt -recursive и terraform validate
  • Использовать линтер tflint
  • В командных проектах — удаленный backend и CI пайплайн с manual step на apply

20 Пайплайны в машинном обучении (MLOps)

20.1 Что такое MLOps

MLOps — это расширение DevOps практик на ML-модели. Цель — повторяемость, версионирование, автоматизация, мониторинг, коллаборация.

По статистике только 15% ML-проектов доходят до production. MLOps повышает эти шансы.

Основные проблемы без MLOps:

  • Notebook hell — модель живет только в Jupyter, непригодна для production
  • Training-serving skew — данные при обучении ≠ данным в production
  • Невоспроизводимость
  • Ручной деплой, нет CI/CD
  • Нет мониторинга
  • Data leakage (утечка целевой переменной в признаки)
  • Pipeline jungles — запутанные зависимости между скриптами

20.2 Уровни зрелости MLOps

Уровень Описание
0 Ручные процессы, DS работают в ноутбуках, нет CI/CD
1 автоматизация обучения
2 CI/CD для моделей
3 Полная автономия (автоматическое переобучение, A/B тесты, откат)

  1. Определение бизнес-задач
  2. Сбор данных и EDA (Exploratory Data Analysis)
  3. Подготовка данных (clean, transform)
  4. Разработка модели
  5. Упаковка модели
  6. Деплой
  7. Мониторинг
  8. (обратно к пункту 1)

Артефакты каждого этапа становятся входом для следующего.

Роли в команде:

  • Data Engineer — пайплайны данных
  • Data Scientist — EDA, эксперименты, baseline модели
  • ML Engineer — упаковка, деплой, CI/CD, мониторинг
  • ML Platform Architect — инфраструктура

20.4 Компоненты ML-пайплайна

  • Data validation (проверка схемы, распределений)
  • Preprocessing
  • Feature engineering
  • Data splitting (train/val/test)
  • Model training
  • Model evaluation
  • Model validation (бизнес-метрики, QA)
  • Model registration (в Model Registry)
  • Model deployment

20.5 Инструменты оркестрации пайплайнов

  • Apache Airflow — самый популярный (DAG на Python)
  • Prefect — более современный, динамические DAG
  • Dagster — с упором на data quality
  • Luigi — старый, от Spotify

20.6 Трекинг ML-экспериментов — MLflow

MLflow Components: - Tracking — логирование параметров, метрик, артефактов (моделей, графиков) - Projects — воспроизводимая упаковка кода (conda, Docker) - Models — стандартный формат моделей (для деплоя) - Registry — централизованное хранение моделей

20.7 Обучение моделей в пайплайне

С ростом моделей и датасетов используют:

  • Параллелизация данных (data parallel) — данные разбиваются на части, каждая обучается на своем устройстве, градиенты усредняются
  • Параллелизация моделей (model parallel) — модель разрезается на части, каждая часть на своем устройстве

Hyperparameter Tuning (от ручного к автоматическому): - GridSearch — экспоненциально дорогой - RandomSearch — случайные точки, эффективнее grid при высокой размерности - Байесовская оптимизация (строит вероятностную модель метрики) - Hyperband / ASHA — ранняя остановка плохих экспериментов

Рекомендация: Optuna (TPE sampler + MedianPruner)

20.8 Model Registry (реестр моделей)

Хранит:

  • Историю версий моделей
  • Артефакты (веса)
  • Метданные (метрики, гиперпараметры)
  • Ссылки на датасет
  • Статус (Staging, Production, Archived)

Паспорт модели — обязателен, иначе модель — черный ящик.

20.9 CI/CD vs Continuous Training

  • CI — тестирование кода, данных, моделей
  • CD — деплой модели в production (blue/green, canary)
  • Continuous Training — автоматическое переобучение при дрифте

Стратегии деплоя (от безопасных к рискованным): - Canary (1% → 10% → 100% трафика) - Blue/Green (полный switch) - A/B тестирование

Model serving frameworks: TensorFlow Serving, TorchServe, Seldon, BentoML, KServe

20.10 Мониторинг моделей в production

Типы дрифта:

  • Data drift — изменились входные данные (распределение признаков)
  • Label drift — изменилось распределение целевой переменной
  • Concept drift — изменилась связь “признак → целевая переменная”
  • Infrastructure drift — изменилось окружение (версии библиотек, железо)

Стек мониторинга:

  • Observability для ML-систем: Evidently, WhyLabs, Arize, Fiddler
  • Операционные метрики: latency, throughput, errors
  • ML-специфичные метрики: data drift, prediction distribution, confidence scores

Алерты и реакция: откат к предыдущей модели, переобучение, оповещение дата саентиста.

20.11 Безопасность и Governance в ML

ML Security Threats:

  • Adversarial attacks (изменение входа, чтобы обмануть модель)
  • Model inversion (восстановление данных обучения)
  • Poisoning (вредоносные данные при обучении)

20.12 Паттерны и антипаттерны MLOps

Паттерны:

  • Champion/Challenger — новая модель получает часть трафика
  • Feature pipeline separation — фичи вычисляются отдельно от обучения
  • Stateless serving — модель не хранит состояние между запросами
  • Feedback loop — сбор реальных меток для дообучения
  • Two-phase prediction — batch + real-time
  • Separate train/serve environments

Антипаттерны:

  • Glue code explosion — бесконечные скрипты-переходники
  • Pipeline jungles — хаотичные зависимости
  • Prototype leakage — Jupyter notebook в production
  • Training-serving skew — разная обработка данных
  • Dead experimental code paths
  • Undeclared consumers — никто не знает, кто юзает эту модель

20.14 Дополнение: SSH в CI/CD пайплайнах

  • SSH-ключ не коммитится, хранится в CI Secrets как переменная окружения.
  • Клиент SSH подключается к заранее развернутому серверу.
  • Используется для деплоя на удаленные машины или проброса портов (ssh -L) до пода в Kubernetes.

Пример использования в пайплайне:

echo "$SSH_PRIVATE_KEY" > private.key
chmod 600 private.key
ssh -i private.key user@server "docker pull myapp && docker-compose up -d"
